2 Scenario YAML Format

How to define a scenario that the system can execute

YAML 
# scenarios/export_limit.yaml

id: export_limit
name: Export Limit
description: Check or modify the site export limit.

training_phrases:                          # Seed phrases (5-10 is enough)
  - "what is the export limit"
  - "show export limit"
  - "export limit of the site"
  - "how many records can I export"
  - "check export limit"

steps:
  - action: run_script                    # Step 1: Run a Python script
    script: get_export_limit.py
    display_message: "Current export limit:"

  - action: ask_choice                    # Step 2: Ask a question
    label: ask_again                       # Label for goto target
    question: "Would you like to change it?"
    options:
      - label: "Yes"
        value: "yes"
      - label: "No"
        value: "no"
    branches:                                  # Different paths per answer
      yes:
        - action: ask_input                # Ask for free-text input
          question: "What should the new limit be?"
          input_key: new_limit

        - action: run_script               # Run script with user input
          script: set_export_limit.py
          args:
            new_limit: "$input.new_limit"    # Variable from ask_input
          display_message: "Export limit updated:"

        - action: goto                     # Loop back to the choice
          target: ask_again                  # Jump to labeled step

      no:
        - action: end
          message: "No changes made."

Step Types

  • ● run_script — Execute a Python script and show the result
  • ● ask_choice — Ask Yes/No or multiple choice (with branches)
  • ● ask_input — Ask for free-text input (store as variable)
  • ● goto — Jump to a labeled step (enables looping flows)
  • ● end — End the scenario with a closing message

Key Features

  • ▶ Branching: ask_choice branches lead to different step sequences
  • ▶ Looping: goto + label enables cycles (e.g., "ask again")
  • ▶ Variables: $input.key_name passes user answers to scripts
  • ▶ Labels: Any step can have a label as a goto target
  • ▶ Simple: Non-developers can read and edit YAML files
3 Python Scripts

The executable actions that scenarios run

Script Contract

  • ▶ Scripts live in backend/scripts/
  • ▶ Accept args via --args '{"key": "val"}'
  • ▶ Print result to stdout (text or JSON)
  • ▶ Exit 0 = success, non-zero = error
  • ▶ 30-second timeout enforced
  • ▶ Runs in sandboxed subprocess (isolated)

Security

  • ▶ Only scripts listed in YAML are executable
  • ▶ Path traversal blocked (can't escape scripts/)
  • ▶ Only .py files allowed
  • ▶ Minimal environment (no secrets exposed)
  • ▶ shell=False (no shell injection)

Technical Details

  • ▶ Embedding: all-MiniLM-L6-v2 (22MB, CPU-friendly)
  • ▶ Classifier: MLPClassifier (256, 128, 64), ReLU
  • ▶ Confidence threshold: 0.4 minimum
  • ▶ Similarity threshold: 0.45 cosine (OOD gate)
  • ▶ Training time: ~30s for 50K+ phrases

Out-of-Distribution Detection

  • ▶ Problem: MLP always picks a class, even for irrelevant input
  • ▶ Solution: Cosine similarity to per-class centroids
  • ▶ Centroids: Mean embedding of all training phrases per scenario
  • ▶ Gate: If max similarity < 0.45, classify as "unknown"
  • ▶ Result: "hello how are you" → UNKNOWN (0.17 similarity)

Two-Stage Classification

  • 1 OOD Gate: Cosine similarity between input embedding and per-class centroids
  • 2 If max similarity < 0.45 → UNKNOWN (skip MLP entirely)
  • 3 MLP Predict: MLPClassifier outputs probability distribution over scenarios
  • 4 If max probability < 0.4 → UNKNOWN
  • 5 Otherwise → return matched scenario ID + confidence

Example Predictions

  • ✅ "what is the export limit" → export_limit (1.000)
  • ✅ "battery charge level" → what_is_soc_of_battery (1.000)
  • ✅ "check user account" → user_account_status (0.977)
  • ❌ "hello how are you" → UNKNOWN (sim: 0.171)
  • ❌ "what is the weather" → UNKNOWN (sim: 0.082)
  • ❌ "tell me a joke" → UNKNOWN (sim: 0.077)

Session State Machine

  • ● IDLE — Waiting for user input, no active scenario. NLP classifies next message.
  • ● AWAITING_CHOICE — Asked a Yes/No question, waiting for user to pick an option.
  • ● AWAITING_INPUT — Asked for free-text input (e.g., "new limit value?").
  • ● EXECUTING — Running a script, will send result when done.

2 Sentence Embeddings

How text becomes numbers that a neural network can understand

From Words to Vectors

✍ Input Text

"What is the export limit?"

⬇

Tokenizer

Split into sub-word tokens: ["what", "is", "the", "export", "limit", "?"]
Map each token to an integer ID from vocabulary

⬇

Transformer Encoder (6 Layers)

Each token attends to every other token (self-attention).
"export" near "limit" produces different embedding than "export" near "goods".
Context determines meaning.

⬇

Mean Pooling

Average all token vectors into a single sentence-level vector.
Result: one 384-dimensional vector that captures the full sentence meaning.

⬇

384-Dimensional Vector

[0.0342, -0.1281, 0.0893, ..., 0.0571] — 384 floating point numbers

What is all-MiniLM-L6-v2?

  • ▶ Pre-trained model from Hugging Face (22MB, lightweight)
  • ▶ 6 Transformer layers (L6) with 384 hidden dimensions
  • ▶ Trained on 1 billion+ sentence pairs for semantic similarity
  • ▶ Key property: Similar sentences → similar vectors (close in space)
  • ▶ Runs on CPU in ~5ms per sentence (no GPU needed)
  • ▶ Vocabulary: 30,522 sub-word tokens (WordPiece tokenizer)

Why Embeddings Work

  • ▶ Semantic space: Words with similar meaning are nearby
  • ▶ "export limit" and "export cap" → vectors are very close (cosine ≈ 0.95)
  • ▶ "export limit" and "battery charge" → vectors are far apart (cosine ≈ 0.15)
  • ▶ Typo-resistant: Sub-word tokenization handles "eksport" → "eks" + "port"
  • ▶ Case-insensitive: Model normalizes internally
  • ▶ Language understanding: "how many can I export" maps near "export limit"

Cosine Similarity — Measuring Closeness

Cosine similarity measures the angle between two vectors. It ranges from -1 (opposite) to 1 (identical direction). In embedding space, semantically similar sentences have cosine similarity close to 1.0. 

# How cosine similarity works:
#                    A · B          (dot product of vectors)
# cos(A, B) = ───────────────────
#              ||A|| × ||B||       (product of magnitudes)

Example embeddings (simplified to 4 dimensions):

"export limit"     →  [0.8, 0.3, 0.1, 0.2]
"export cap"       →  [0.79, 0.31, 0.09, 0.22]   cosine = 0.998  # Nearly identical meaning
"battery charge"   →  [0.1, 0.2, 0.9, 0.1]    cosine = 0.312  # Very different meaning
"hello world"      →  [0.4, 0.5, 0.3, 0.6]    cosine = 0.171  # Unrelated — OOD detected

What is an MLP (Multi-Layer Perceptron)?

  • ▶ Feedforward neural network: data flows in one direction (input → output)
  • ▶ Fully connected: every neuron connects to every neuron in the next layer
  • ▶ Total parameters: 384×256 + 256×128 + 128×64 + 64×N + biases ≈ 140K weights
  • ▶ Each neuron computes: output = ReLU(W · input + bias)
  • ▶ Learning: Backpropagation adjusts weights to minimize classification error

Key Components Explained

  • ● ReLU Activation: f(x) = max(0, x). Introduces non-linearity — allows the network to learn complex decision boundaries, not just straight lines
  • ● Softmax Output: Converts final layer scores into probabilities that sum to 1.0. Highest probability = predicted scenario
  • ● Early Stopping: Monitor validation loss during training; stop when it starts increasing to prevent overfitting
  • ● Validation Split: 15% of data held out during training to detect overfitting

How Training Works (Step by Step)

Step 1: Prepare Training Data

10,000+ phrases per scenario, each labeled with its scenario_id.
Example: ("what is the export limit", "export_limit")

⬇

Step 2: Encode All Phrases

SentenceTransformer converts each phrase to a 384-dim vector.
30,000 phrases → matrix of shape [30000, 384]

⬇

Step 3: Forward Pass

Feed each embedding through the network. Compare predicted probabilities to the true label.
Calculate loss (cross-entropy): how wrong was the prediction?

⬇

Step 4: Backpropagation

Compute gradients: how should each weight change to reduce the error?
Update weights using Adam optimizer (adaptive learning rate).

⬇

Step 5: Repeat (up to 500 epochs)

Each epoch = one pass through all training data.
Early stopping: if validation accuracy doesn't improve for 10 epochs, stop.

⬇

Step 6: Save Model + Centroids

Save trained weights to intent_classifier.joblib.
Compute mean embedding per class → centroids.npy for OOD detection.

4 Out-of-Distribution (OOD) Detection

How SmartChat knows when a query doesn't match ANY scenario

The Problem: Softmax Always Picks Something

A neural network with softmax output always assigns probabilities that sum to 1.0. Even for completely unrelated input, one class will get the highest probability — often with high confidence. Without OOD detection, "hello how are you" would be classified as a scenario with 96% confidence! 

# WITHOUT OOD detection (broken!):
"hello how are you" → MLP softmax → [0.02, 0.96, 0.02]
                                          export  soc_battery  account
                                          # WRONG! 96% confident in battery scenario for "hello"

# WITH OOD detection (fixed!):
"hello how are you" → cosine similarity to centroids → max = 0.171
                         0.171 < 0.45 threshold → UNKNOWN (skip MLP entirely)
OOD Detection: Centroid-Based Gating

During Training: Compute Centroids

For each scenario, average all its training phrase embeddings into one "centroid" vector.
3 scenarios → 3 centroid vectors (each 384-dim)

⬇

At Runtime: Encode User Query

"hello how are you" → 384-dim embedding vector

⬇

Compute Cosine Similarity to All Centroids

Compare query embedding against each centroid:
similarity_export = 0.152 | similarity_battery = 0.171 | similarity_account = 0.098

⬇

Max Similarity ≥ 0.45?

max(0.152, 0.171, 0.098) = 0.171 < 0.45

⬇
✅ ≥ 0.45 — In Distribution
⬇

Pass to MLP Classifier

Query is close enough to known scenarios.
Use neural network to determine which one.

❌ < 0.45 — Out of Distribution
⬇

UNKNOWN → LLM Fallback

Query is too far from all known scenarios.
Route to Claude API for an AI-generated response.

Why Centroids?

  • ▶ Cheap: One 384-dim vector per class (stored in centroids.npy)
  • ▶ Fast: One matrix multiply at inference (<1ms)
  • ▶ Intuitive: Centroid = "average meaning" of all training phrases for a scenario
  • ▶ Threshold tunable: 0.45 is conservative; lower = more strict, higher = more lenient
  • ▶ No extra model needed: Reuses the same embeddings from training

Real-World OOD Examples

  • ✅ "what is the export limit" → sim = 0.92 → IN (export_limit)
  • ✅ "battery charge level" → sim = 0.88 → IN (soc_battery)
  • ✅ "check user account" → sim = 0.61 → IN (account_status)
  • ❌ "hello how are you" → sim = 0.171 → OOD
  • ❌ "what is the weather" → sim = 0.082 → OOD
  • ❌ "tell me a joke" → sim = 0.077 → OOD
  • ❌ "reset my password" → sim = 0.342 → OOD

Classification Code (intent_classifier.py)

Python 
def predict(self, text: str) -> tuple[str | None, float]:
    # Step 1: Encode text to embedding
    embedding = self.embedder.encode_single(text).reshape(1, -1)

    # Step 2: OOD Gate — cosine similarity to class centroids
    if self.centroids is not None:
        norm_emb = embedding / np.linalg.norm(embedding, axis=1, keepdims=True)
        norm_cent = self.centroids / np.linalg.norm(self.centroids, axis=1, keepdims=True)
        similarities = (norm_emb @ norm_cent.T)[0]        # Cosine similarity
        max_similarity = float(similarities.max())
        if max_similarity < self.similarity_threshold:     # 0.45 default
            return None, max_similarity                     # UNKNOWN — OOD

    # Step 3: MLP Classification
    probabilities = self.clf.predict_proba(embedding)[0]
    max_idx = probabilities.argmax()
    confidence = float(probabilities[max_idx])
    scenario_id = self.label_encoder.inverse_transform([max_idx])[0]

    if confidence < self.confidence_threshold:           # 0.4 default
        return None, confidence                           # UNKNOWN — low confidence
    return scenario_id, confidence                        # MATCH!

Performance (per query)

  • ▶ Embedding: ~5ms (SentenceTransformer on CPU)
  • ▶ OOD check: <0.1ms (one matrix multiply)
  • ▶ MLP predict: <0.5ms (forward pass through 3 layers)
  • ▶ Total classification: ~6ms end-to-end
  • ▶ LLM fallback: ~1-3 seconds (Claude API call, only when OOD)

Two Thresholds

  • ● Similarity threshold (0.45): "Is this query even related to any scenario?"
    Low = strict (rejects more), High = lenient (accepts more)
  • ● Confidence threshold (0.4): "Is the MLP sure which scenario it is?"
    Safety net after OOD passes, catches ambiguous queries
6 LLM Fallback (Claude API)

When the neural network says "I don't know", a large language model provides an intelligent response

How It Works

  • 1 Classifier returns None (unknown query)
  • 2 Flow engine checks if LLM fallback is available
  • 3 Send user's message to Claude API with system prompt
  • 4 Claude generates a helpful, context-aware response
  • 5 Response sent as "ai_response" type (purple bubble)
  • 6 User stays in IDLE state (no scenario activated)

Why Two AI Layers?

  • ▶ Layer 1 (Neural Network): Fast (~6ms), deterministic, domain-specific. Perfect for known scenarios. Cannot improvise.
  • ▶ Layer 2 (LLM / Claude): Slow (~2s), generative, general-purpose. Handles anything. Expensive per call.
  • ▶ Best of both worlds: Use the fast, cheap model for known queries. Only call the powerful LLM when needed.
  • ▶ Cost optimization: ~95% of queries hit scenarios (free). ~5% hit LLM (paid API call).

7 Design Decisions & Architecture Rationale

Why SmartChat is designed this way — the tradeoffs and reasoning

Why MLP over a Transformer Classifier?

  • ▶ Embeddings already capture semantics. The SentenceTransformer does the hard language understanding. MLP just draws decision boundaries.
  • ▶ Fast to train: 30 seconds vs 30+ minutes for fine-tuning a Transformer
  • ▶ Small model: ~1.6MB vs 100MB+ for a Transformer classifier
  • ▶ Easy to retrain: Add a scenario, retrain in seconds. No GPU needed.
  • ▶ Scikit-learn: No PyTorch/TensorFlow dependency for inference

Why Augmentation with Claude API?

  • ▶ User writes 5 phrases, system needs 10,000 for good coverage
  • ▶ Claude understands intent: Generates meaningful variations, not random noise
  • ▶ Covers edge cases: Typos, slang, formal, casual, different languages
  • ▶ Cached: Pay once per scenario, reuse forever
  • ▶ Fallback: Basic augmentation (case changes, punctuation) when no API key

Why Centroid OOD vs. Other Methods?

  • ▶ No extra model: Just a numpy array (3×384 for 3 scenarios)
  • ▶ Alternative: Train an "other" class — requires curating negative examples, hard to maintain
  • ▶ Alternative: Temperature scaling — still doesn't reject truly OOD inputs
  • ▶ Alternative: Mahalanobis distance — more complex, covariance matrix needed
  • ▶ Centroids are the sweet spot: Simple, effective, explainable, fast

Why WebSocket for Chat?

  • ▶ Bidirectional: Server pushes responses without client polling
  • ▶ Multi-step scenarios: One user message can trigger multiple server responses (typing indicator, script result, question)
  • ▶ Real-time: Typing indicators, instant results
  • ▶ Stateful: Session maintained throughout the conversation
  • ▶ Auto-reconnect: Frontend hook handles disconnections gracefully

Overall System Design Philosophy

Scenario-Driven

Instead of a general AI chatbot, SmartChat uses predefined workflows. Each scenario is a deterministic flow: run scripts, ask questions, branch on answers. This ensures reliable, predictable behavior for support operations where accuracy matters more than creativity.

AI at the Edges

AI is used at two points: intent classification (understanding what the user wants) and fallback responses (when no scenario matches). The core flow engine is deterministic — no AI hallucination in the actual business logic. Scripts run real code, not generated code.

Progressive Complexity

Simple scenarios = simple YAML. Complex scenarios use branching and looping with goto. The visual flow editor makes it accessible to non-developers while the underlying YAML format keeps it version-controllable and diff-friendly.
