2 Scenario YAML Format

How to define a scenario that the system can execute

YAML 
# scenarios/export_limit.yaml

id: export_limit
name: Export Limit
description: Check or modify the site export limit.

training_phrases:                          # Seed phrases (5-10 is enough)
  - "what is the export limit"
  - "show export limit"
  - "export limit of the site"
  - "how many records can I export"
  - "check export limit"

steps:
  - action: run_script                    # Step 1: Run a Python script
    script: get_export_limit.py
    display_message: "Current export limit:"

  - action: ask_choice                    # Step 2: Ask a question
    question: "Would you like to change it?"
    options:
      - label: "Yes"
        value: "yes"
      - label: "No"
        value: "no"
    branches:                                  # Different paths per answer
      yes:
        - action: ask_input                # Ask for free-text input
          question: "What should the new limit be?"
          input_key: new_limit

        - action: run_script               # Run script with user input
          script: set_export_limit.py
          args:
            new_limit: "$input.new_limit"    # Variable from ask_input
          display_message: "Export limit updated:"

      no:
        - action: end
          message: "No changes made."

Step Types

  • ● run_script — Execute a Python script and show the result
  • ● ask_choice — Ask Yes/No or multiple choice (with branches)
  • ● ask_input — Ask for free-text input (store as variable)
  • ● end — End the scenario with a closing message

Key Features

  • ▶ Branching: ask_choice branches lead to different step sequences
  • ▶ Variables: $input.key_name passes user answers to scripts
  • ▶ Nesting: Branches can contain more ask_choice steps
  • ▶ Simple: Non-developers can read and edit YAML files
3 Python Scripts

The executable actions that scenarios run

Script Contract

  • ▶ Scripts live in backend/scripts/
  • ▶ Accept args via --args '{"key": "val"}'
  • ▶ Print result to stdout (text or JSON)
  • ▶ Exit 0 = success, non-zero = error
  • ▶ 30-second timeout enforced
  • ▶ Runs in sandboxed subprocess (isolated)

Security

  • ▶ Only scripts listed in YAML are executable
  • ▶ Path traversal blocked (can't escape scripts/)
  • ▶ Only .py files allowed
  • ▶ Minimal environment (no secrets exposed)
  • ▶ shell=False (no shell injection)

Example: get_export_limit.py

Python 
import json

def main():
    # Query database or API

        "export_limit": 1000,

Example: set_export_limit.py

Python 
import json, argparse

def main():
    parser = argparse.ArgumentParser()
    parser.add_argument("--args", required=True)
    parsed = parser.parse_args()
    args = json.loads(parsed.args)

    new_limit = args["new_limit"]
    # Update database or API
    result = {
        "status": "success",
        "new_export_limit": int(new_limit)
    }
    print(json.dumps(result))

if __name__ == "__main__":
    main()

Covers typos, capitalization, slang, formal, casual, questions, commands

Claude API Augmentation Prompt

// Sent to Claude for EACH scenario

"""
You are a training data generator. I have a chatbot scenario called
"{scenario_name}" with these seed phrases:

{seed_phrases}

Generate exactly 10,000 unique variations. Include:
- Different phrasings (formal, casual, short, long)
- UPPERCASE, lowercase, Mixed Case, aLL cAPS
- Common typos and misspellings
- With and without punctuation
- Slang and abbreviations
- Different word orders
- Questions vs statements
- With extra words / filler words
- Multi-language hints (Frenglish, etc.)

Return ONLY the phrases, one per line. No numbering.
"""

Why Claude Augmentation?

  • ▶ You write 5 — Claude generates 10,000 — massive training data
  • ▶ Covers typos: "exprt limit", "eksport lmit"
  • ▶ Covers caps: "EXPORT LIMIT", "Export Limit"
  • ▶ Covers phrasing: formal, casual, short, verbose
  • ▶ Cached: saved to JSON, no repeated API calls

How It Works

  • 1 Read seed phrases from YAML
  • 2 Send to Claude API with augmentation prompt
  • 3 Parse response (one phrase per line)
  • 4 Save to augmented_data/{scenario_id}.json
  • 5 Next time: load from cache (skip API call)

Technical Details

  • ▶ Embedding: all-MiniLM-L6-v2 (22MB, CPU-friendly)
  • ▶ Classifier: MLPClassifier (256, 128, 64), ReLU
  • ▶ Confidence threshold: 0.4 minimum
  • ▶ Training time: ~30s for 50K+ phrases

Session State Machine

  • ● IDLE — Waiting for user input, no active scenario. NLP classifies next message.
  • ● AWAITING_CHOICE — Asked a Yes/No question, waiting for user to pick an option.
  • ● AWAITING_INPUT — Asked for free-text input (e.g., "new limit value?").
  • ● EXECUTING — Running a script, will send result when done.

6 System Architecture

Backend and frontend component breakdown

Backend (FastAPI / Python)

  • ▶ main.py — App entry, CORS, startup/shutdown
  • ▶ api/chat.py — WebSocket chat endpoint
  • ▶ api/scenarios.py — REST: list, get, reload
  • ▶ api/training.py — REST: train, status
  • ▶ core/flow_engine.py — Scenario state machine
  • ▶ core/script_executor.py — Sandboxed runner
  • ▶ core/scenario_loader.py — Parse YAML
  • ▶ core/session_manager.py — Session tracking
  • ▶ nlp/augmenter.py — Claude API augmentation
  • ▶ nlp/embedder.py — SentenceTransformer
  • ▶ nlp/trainer.py — Train MLPClassifier
  • ▶ nlp/intent_classifier.py — Predict intent

Frontend (React / TypeScript)

  • ▶ ChatContainer — Main orchestrator
  • ▶ MessageList — Scrollable messages
  • ▶ MessageBubble — Message display
  • ▶ QuickReplyButtons — Choice buttons
  • ▶ ScriptResultCard — Script output
  • ▶ useWebSocket — WS + reconnect
  • ▶ useChatSession — Message state
  • ▶ Sidebar — Scenario list
